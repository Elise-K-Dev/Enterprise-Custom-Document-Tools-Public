# 현재 아키텍처 및 코드 구현 설명

작성 기준일: 2026-04-29  
분석 범위: 현재 레포지토리의 실행 구조, 서비스 구성, 주요 API, 데이터 흐름, 코드 구현 방식  
제외 범위: `항만시스템 문제점 분석.pdf`의 문제점 본문 및 별도 항만시스템 분석 내용

## 1. 개요

현재 시스템은 Open WebUI를 사용자 접점으로 두고, 내부 문서 검색, 구매 품의 문서 생성, 재고/입출고 스냅샷 조회, Markdown 기반 보고서 렌더링, 외부 웹페이지 본문 추출을 각각 독립 서비스로 분리한 구조이다.

전체적인 목적은 다음과 같다.

- 사용자가 Open WebUI에서 자연어로 요청한다.
- Open WebUI가 등록된 OpenAPI 도구를 호출한다.
- 각 도구 서비스가 문서 검색, 재고 조회, DOCX/PDF/XLSX 생성, 웹 본문 추출을 수행한다.
- 생성 파일은 서비스 내부 출력 디렉터리에 저장되고, 제한 시간 다운로드 토큰이 붙은 URL로 반환된다.

핵심 특징은 “대화형 AI 화면은 하나, 실제 업무 기능은 여러 내부 서비스”라는 점이다. Rust 서비스가 구매 품의와 재고 문서 생성의 중심 게이트웨이 역할을 하고, Python 계열 서비스들이 검색/렌더링/웹 추출을 맡는다.

## 2. 전체 아키텍처

```text
사용자
  |
  v
Open WebUI (:3000)
  |
  +-- document_generation_tools
  |     |
  |     v
  |   Rust document-service (:8001)
  |     |-- 구매 품의 DOCX 생성
  |     |-- 재고/부족 품목 조회
  |     |-- 입고/재고/출고 Excel 배치 처리
  |     |-- Markdown 렌더러 프록시 호출
  |     v
  |   markdown-pdf-service (:8003)
  |     |-- Markdown -> PDF
  |     |-- 대화/보고서 -> DOCX
  |     |-- 대화/보고서 -> XLSX
  |
  +-- document_search
  |     |
  |     v
  |   parser-service (:8002)
  |     |-- 내부 Markdown 문서 검색
  |     |-- 권한별 문서 필터링
  |     |-- BM25 검색 및 LLM 답변 생성
  |     |-- 문서 필드 초안 생성
  |
  +-- web_fetch
        |
        v
      web-service (:8004)
        |-- URL 본문 추출
        |-- SPA 렌더링 후 Markdown 변환
```

## 3. 서비스별 역할

| 서비스 | 포트 | 주요 파일 | 역할 |
|---|---:|---|---|
| Open WebUI | 3000 | `open-webui/*.json` | 사용자 UI 및 OpenAPI 도구 등록 |
| Rust document-service | 8001 | `rust-service/src/main.rs` | 통합 문서 생성 게이트웨이 |
| Legacy engine | 내부 모듈 | `rust-service/src/legacy_engine.rs` | Excel 배치 처리, 재고 스냅샷, 구매 문서 대량 생성 |
| Legacy port | 내부 모듈 | `rust-service/src/legacy_port.rs` | DOCX 템플릿 로딩 및 치환 |
| parser-service | 8002 | `python-service/app/main.py` | 내부 문서 검색, 권한 처리, 필드 초안 생성 |
| markdown-pdf-service | 8003 | `markdown-pdf-service/app/main.py` | PDF/DOCX/XLSX 렌더링 |
| web-service | 8004 | `web-service/app/main.py` | 외부 URL 본문 추출 |

## 4. Docker 및 실행 구조

기본 실행 구성은 `docker-compose.yml`에 정의되어 있다.

- `open-webui`는 사용자 화면을 제공한다.
- `document-service`는 `./rust-service/DB`를 `/app/DB`로 마운트한다.
- `parser-service`는 전처리된 Markdown 문서, 카탈로그, 사용자 정보를 읽기 전용으로 마운트한다.
- `markdown-pdf-service`는 생성 파일을 `./markdown-pdf-service/output`에 저장한다.
- `web-service`는 crawl4ai 기반으로 외부 URL을 가져온다.

개발 환경에서는 `docker-compose.host.yml`을 오버레이로 사용하여 모든 서비스를 host network로 올릴 수 있다. 이 경우 Open WebUI가 내부 Docker 서비스명 대신 `127.0.0.1:8001`, `127.0.0.1:8002`, `127.0.0.1:8003`, `127.0.0.1:8004` 형태로 도구 서버에 접근한다.

중요한 공통 환경 변수는 다음과 같다.

| 변수 | 용도 |
|---|---|
| `PORT_PROJECT_INTERNAL_TOKEN` | 내부 도구 호출 인증 토큰 |
| `PORT_PROJECT_DOWNLOAD_TOKEN_TTL_SECONDS` | 다운로드 토큰 유효 시간 |
| `DOCUMENT_SERVICE_PUBLIC_BASE_URL` | Rust 서비스가 사용자에게 반환할 공개 URL 기준 |
| `MARKDOWN_PDF_INTERNAL_BASE_URL` | Rust 서비스가 렌더러를 호출할 내부 URL |
| `PORT_PROJECT_LEGACY_WORKDIR` | 구매/재고 레거시 작업 디렉터리 |
| `DOCUMENT_FILLER_API_URL` | parser-service가 호출하는 LLM Chat Completions API |
| `DOCUMENT_FILLER_MODEL_ID` | 필드 생성/검색 답변에 사용할 모델명 |

## 5. Open WebUI 도구 구성

Open WebUI에는 JSON 파일로 도구가 등록된다.

- `openwebui-rust-tools.json`: `document_generation_tools`
- `openwebui-python-tools.json`: `document_search`
- `openwebui-markdown-pdf-tools.json`: Markdown 렌더러 직접 등록용
- `openwebui-web-tools.json`: 웹페이지 가져오기 도구

현재 권장 흐름은 Rust 서비스의 `document_generation_tools`를 통합 진입점으로 쓰는 방식이다. 구매 품의, 재고 조회, 보고서 파일 생성, PDF/Word/Excel 내보내기를 Rust 서비스에서 한 번 감싸고, 필요 시 markdown-pdf-service로 프록시한다.

## 6. Rust document-service 구현

Rust 서비스는 Axum 기반 HTTP API 서버이다. 진입점은 `rust-service/src/main.rs`이며, `app_router()`에서 주요 라우트를 등록한다.

주요 API는 다음과 같다.

| API | 기능 |
|---|---|
| `GET /health` | 상태 확인 |
| `GET /openapi.json` | Open WebUI 도구 스펙 제공 |
| `POST /document/create` | 구매 품의 문서 작성 세션 생성 |
| `POST /document/fill` | 사용자 추가 입력을 세션 필드에 병합 |
| `POST /document/export` | 필드 기반 문서 내보내기 |
| `POST /document/legacy/run` | 입고/재고/출고 배치 처리 실행 |
| `POST /document/legacy/package` | 구매 품의 DOCX 대량 생성 후 ZIP 반환 |
| `GET /document/legacy/shortages` | 부족 재고 품목 조회 |
| `GET /document/legacy/items` | 전체 재고 품목 조회 |
| `GET /document/legacy/items/report` | 재고 현황 CSV 보고서 생성 |
| `GET /document/legacy/item-context` | 단일 품목 문서 작성용 컨텍스트 생성 |
| `POST /document/legacy/item-export` | 단일 품목 DOCX 생성 |
| `POST /document/legacy/item-approve` | 품목 승인 후 최종 DOCX 생성 |
| `POST /document/pdf/render` | Markdown PDF 렌더러 프록시 |
| `POST /document/chat/docx` | 채팅/보고서 DOCX 렌더러 프록시 |
| `POST /document/chat/xlsx` | 채팅/보고서 XLSX 렌더러 프록시 |

### 6.1 상태 관리

`AppState`는 다음 상태를 공유한다.

- 템플릿 정의 목록
- 대화형 문서 작성 세션
- 다운로드 토큰 grant
- 레거시 작업 디렉터리
- 공개 URL 기준값
- Markdown 렌더러 내부 URL
- upstream 호출용 HTTP client

세션은 `RwLock<HashMap<String, SessionState>>`로 메모리에 저장된다. `create_document`에서 UUID 기반 세션을 만들고, `fill_document`에서 기존 세션 필드와 새 입력을 병합한다.

### 6.2 인증 및 다운로드 보호

Rust 서비스는 Open WebUI가 넘겨주는 내부 토큰과 사용자 헤더를 확인한다.

- 내부 호출 토큰: `x-port-project-internal-token`
- 사용자 이메일: `x-openwebui-user-email`
- 사용자 ID: `x-openwebui-user-id`
- 사용자 이름: `x-openwebui-user-name`

파일 다운로드는 바로 경로를 노출하지 않고, 서비스 메모리에 다운로드 grant를 발급한 뒤 `token` 쿼리로 검증한다. 다운로드 시에는 요청 파일이 허용된 출력 디렉터리 내부에 있는지도 다시 확인한다.

### 6.3 구매 품의 문서 작성 흐름

대화형 문서 작성은 다음 순서로 진행된다.

1. 사용자가 품목명, 수량, 업체, 사유 등을 자연어로 입력한다.
2. `create_document`가 텍스트에서 필드를 추출한다.
3. 품목명 또는 품번이 있으면 `stock_in_out_monthly.json` 스냅샷에서 재고/입출고 문맥을 보강한다.
4. 필수 필드 누락 여부를 계산한다.
5. 다음 질문과 미리보기 텍스트를 반환한다.
6. 사용자가 추가 답변을 주면 `fill_document`가 기존 필드와 새 필드를 병합한다.
7. 최종적으로 `export_document`, `item-export`, `item-approve`가 DOCX를 생성한다.

현재 템플릿 ID는 `purchase_request` 중심으로 구성되어 있다. 일반 보고서 PDF나 채팅 내보내기는 구매 품의 템플릿이 아니라 Markdown 렌더링 계열 API를 사용한다.

## 7. Legacy engine 구현

`rust-service/src/legacy_engine.rs`는 입고/재고/출고 Excel을 읽고, 통합 스냅샷 JSON과 구매 품의 DOCX를 생성하는 배치 엔진이다.

주요 처리 순서는 다음과 같다.

```text
DB/input
  |
  |-- 입고 내역 Excel
  |-- 재고 현황 Excel
  |-- 출고 내역 Excel
  v
legacy_engine
  |
  |-- 파일 탐지 및 안정성 확인
  |-- Excel 첫 시트 읽기
  |-- 헤더 정규화
  |-- 품번/품명 기준 병합
  |-- 입고/출고 수량 합산
  |-- 현재고, 필수재고, 부족수량 계산
  |-- Part_function.json 기반 부품 역할 보강
  |-- stock_in_out_monthly.json 저장
  |-- 구매 대상 DOCX 생성
```

핵심 산출물은 `rust-service/DB/output/stock_in_out_monthly.json`이다. 이후 재고 조회, 부족 품목 조회, 품목별 문서 생성은 원본 Excel을 직접 다시 읽지 않고 이 스냅샷 JSON을 기준으로 수행한다.

### 7.1 재고 계산 기준

스냅샷은 품목 단위로 다음 정보를 가진다.

- 품목명, 품번
- 현재고
- 필수재고
- 입고 합계
- 출고 합계
- 입출고 순증감
- 부족수량
- 입고/출고 일자 목록
- 재고 확인 여부
- 문서 작성용 context

부족 품목 판단은 현재고와 필수재고를 비교하는 방식이다. 재고 데이터가 확정되지 않은 품목은 `inventory_confirmed=false`로 분리되어, 확정 부족 품목과 별도 목록으로 관리된다.

### 7.2 구매 결정 규칙

구매 진행 여부와 템플릿 종류는 `decide_purchase_v2` 규칙으로 결정된다.

- 필수재고가 없거나 0 이하이면 구매 제외
- 현재고가 필수재고 이상이면 구매 제외
- 현재고가 필수재고의 30%를 초과하면 구매 제외
- 단가가 500,000원 이상이면 50만원 이상 템플릿 사용
- 단가가 500,000원 미만이면 50만원 이하 템플릿 사용

교체 이력 유무까지 반영하여 실제 DOCX 템플릿은 네 가지 중 하나가 선택된다.

| 구분 | 교체 이력 있음 | 교체 이력 없음 |
|---|---|---|
| 50만원 이상 | 부품구매요청 + 교체이력 유 | 부품구매요청 + 교체이력 무 |
| 50만원 이하 | 부품구매 + 교체이력 유 | 부품구매 + 교체이력 무 |

## 8. Legacy port DOCX 렌더링 구현

`rust-service/src/legacy_port.rs`는 DOCX 템플릿을 zip 패키지로 읽고, `word/document.xml` 내부 텍스트 토큰을 치환한다.

구현 방식은 다음과 같다.

- DOCX 파일을 ZIP archive로 연다.
- 모든 entry를 메모리에 읽어 `TemplatePackage`로 보관한다.
- `word/document.xml`만 별도로 문자열 처리한다.
- `{{필드명}}` 또는 일부 legacy 토큰을 실제 값으로 치환한다.
- 교체 이력 행이 비어 있으면 해당 표 행을 제거한다.
- 다시 ZIP으로 묶어 DOCX bytes를 반환한다.

이 방식은 Word 템플릿 자체의 스타일, 표, 머리말/꼬리말, 레이아웃을 유지하면서 내용만 바꾸는 데 초점이 있다.

## 9. parser-service 구현

`python-service/app/main.py`는 FastAPI 기반 서비스이다. 역할은 크게 세 가지이다.

### 9.1 내부 문서 검색

`POST /search/query`는 전처리된 Markdown 문서에서 관련 근거를 찾고 답변을 생성한다.

처리 흐름은 다음과 같다.

1. Open WebUI 사용자 헤더 또는 legacy username/password로 사용자 식별
2. 사용자 권한을 `hi_rank` 또는 `low_rank`로 결정
3. `file_catalog.json`에서 권한에 맞는 문서만 필터링
4. LLM을 호출해 질의의 연도, 월, 검색어를 구조화
5. Kiwi 형태소 분석기로 문서를 토큰화
6. BM25로 상위 문서 검색
7. 검색 문서를 context로 묶어 LLM 답변 생성

`low_rank` 사용자는 민감 주제 검색 시 차단될 수 있다. `hi_rank`는 `hi_rank`와 `low_rank` 문서를 모두 볼 수 있고, `low_rank`는 `low_rank` 문서만 볼 수 있다.

### 9.2 문서 필드 초안 생성

`POST /document/fill-fields`는 구매 품의 문서 작성에 필요한 필드 초안을 생성한다.

- 입력: 사용자 질의, 재고/문서 context, 이전 대화, guided fields
- 처리: LLM Chat Completions API 호출
- 출력: `suggested_fields` JSON

이 기능은 Rust 서비스가 만든 품목 컨텍스트와 결합하면, 사람이 직접 채워야 하는 구매 사유, 담당자, 부품 역할 설명 등을 자동 초안화하는 데 사용된다.

### 9.3 간단 Markdown 변환 및 PDF 렌더링

`POST /parser/to-md`는 원문 텍스트를 섹션 단위 Markdown으로 정리한다.  
`POST /render/markdown-pdf`도 존재하지만, 현재 전체 구조에서는 별도 `markdown-pdf-service`가 더 완성된 렌더러 역할을 맡고 있다.

## 10. markdown-pdf-service 구현

`markdown-pdf-service/app/main.py`는 보고서 및 대화 기록을 실제 파일로 렌더링한다.

주요 API는 다음과 같다.

| API | 기능 |
|---|---|
| `POST /render/markdown-pdf` | Markdown 본문을 PDF로 렌더링 |
| `POST /render/chat-docx` | 메시지 또는 transcript를 Word DOCX로 내보내기 |
| `POST /render/chat-xlsx` | 메시지 또는 transcript를 Excel XLSX로 내보내기 |
| `GET /download` | 토큰 검증 후 파일 다운로드 |

PDF 렌더링은 `markdown-it-py`로 Markdown을 HTML로 바꾸고, Playwright Chromium으로 PDF를 출력한다. DOCX는 `python-docx`, XLSX는 `openpyxl`을 사용한다.

문서에는 생성 대상자, 생성 시각, 계정 정보가 메타 정보로 들어간다. 다운로드는 내부 토큰과 사용자 헤더를 확인한 뒤 발급한 임시 download token으로 보호된다.

## 11. web-service 구현

`web-service/app/main.py`는 사용자가 직접 제공한 URL을 가져와 Markdown 본문으로 변환한다.

주요 특징은 다음과 같다.

- `crawl4ai`의 `AsyncWebCrawler` 사용
- Chromium 기반 headless 브라우징
- SPA hydration 대기 시간 설정 가능
- pruning content filter로 본문 중심 Markdown 추출
- 검색 엔진 기능은 제공하지 않음

즉, 이 서비스는 “인터넷에서 검색해줘”가 아니라 “이 URL의 본문을 가져와줘”에 대응한다. 내부 문서 검색은 parser-service의 `document_search`가 담당한다.

## 12. 데이터 흐름 상세

### 12.1 재고 스냅샷 생성 흐름

```text
입고 Excel + 재고 Excel + 출고 Excel
  -> legacy_engine.rs
  -> 품번/품명 기준 병합
  -> 수량/일자/이력 계산
  -> stock_in_out_monthly.json
  -> 부족 품목 조회, 품목 context, 구매 문서 생성에 사용
```

이 구조의 장점은 원본 Excel의 형식 차이나 읽기 비용을 배치 단계에서 흡수하고, 이후 API 요청은 정규화된 JSON만 빠르게 조회한다는 점이다.

### 12.2 단일 품목 구매 문서 생성 흐름

```text
사용자: "A 품목 문서 만들어줘"
  -> /document/legacy/item-context
  -> 스냅샷에서 품목 검색
  -> fields_seed + guided_fields 반환
  -> 필요 시 /document/fill-fields로 초안 생성
  -> /document/legacy/item-approve
  -> 템플릿 선택
  -> DOCX 생성
  -> download_url 반환
```

이 흐름은 “조회 → 초안 → 승인 → 파일 생성”으로 나누어져 있다. 단순 자동 생성보다 사용자가 중간에 확인하고 보강할 수 있는 구조이다.

### 12.3 Markdown 보고서 생성 흐름

```text
사용자 또는 모델이 Markdown 보고서 작성
  -> Rust /document/pdf/render
  -> markdown-pdf-service /render/markdown-pdf
  -> PDF 파일 저장
  -> renderer download token 발급
  -> Rust가 자체 download token으로 다시 감싸서 URL 반환
```

Rust가 렌더러 응답을 다시 감싸는 이유는 Open WebUI에 노출되는 다운로드 URL을 문서 생성 게이트웨이 기준으로 통일하기 위해서이다.

## 13. 보안 및 권한 구조

현재 구현된 보안 장치는 다음과 같다.

- 내부 도구 호출 토큰 검증
- Open WebUI 사용자 헤더 검증
- 등록 사용자만 파일 생성/다운로드 가능
- 다운로드 토큰 TTL 적용
- 다운로드 경로 canonical path 검증
- 검색 문서 권한을 `hi_rank`, `low_rank`로 분리
- low rank 민감 질의 차단 로직

주의할 점도 있다.

- 다운로드 grant는 메모리에 저장되므로 서비스 재시작 시 사라진다.
- 세션 상태도 메모리 기반이므로 재시작 시 문서 작성 세션이 유지되지 않는다.
- 내부 토큰이 유출되면 도구 호출 경계가 약해질 수 있으므로 `.env` 관리가 중요하다.

## 14. 현재 구현의 강점

### 14.1 기능 분리가 명확하다

구매 품의, 문서 검색, 파일 렌더링, 웹 본문 추출이 별도 서비스로 나뉘어 있다. 특정 기능을 수정할 때 전체 시스템을 동시에 건드릴 필요가 적다.

### 14.2 업무 데이터 기준점이 명확하다

재고 조회와 구매 문서 생성의 기준을 `stock_in_out_monthly.json`으로 통일했다. 이 덕분에 API마다 원본 Excel을 다르게 해석하는 문제를 줄일 수 있다.

### 14.3 실제 파일 생성까지 연결되어 있다

단순 텍스트 답변이 아니라 DOCX, PDF, XLSX, ZIP 생성까지 구현되어 있다. Open WebUI 도구 호출 결과가 바로 업무 산출물로 이어진다.

### 14.4 한국어 문서 검색에 맞춘 검색 구성이 있다

parser-service는 Kiwi 형태소 분석기와 BM25를 사용한다. 한국어 문서에서 단순 문자열 검색보다 안정적인 검색 결과를 기대할 수 있다.

## 15. 현재 구현의 한계 및 유의사항

### 15.1 일부 문서/코드 한글 인코딩 표시가 깨져 있다

README 및 일부 Rust/Python 문자열에서 한글이 깨져 보인다. 실행 로직 자체와 별개로, OpenAPI 설명, 사용자 메시지, 템플릿 필드명이 깨져 보일 수 있어 운영 문서성과 유지보수성이 떨어진다.

### 15.2 세션과 다운로드 grant가 메모리 기반이다

단일 프로세스 실행에는 충분하지만, 다중 인스턴스나 재시작이 잦은 운영 환경에서는 세션/토큰 저장소를 Redis 같은 외부 저장소로 분리하는 편이 안정적이다.

### 15.3 레거시 엔진과 포트 모듈에 중복 모델이 있다

`legacy_engine.rs`와 `legacy_port.rs` 모두 `DocumentRow`, 템플릿 선택, DOCX 렌더링과 관련된 구조를 일부 갖고 있다. 지금은 동작상 분리되어 있지만, 장기적으로는 공통 모델을 정리하면 유지보수가 쉬워진다.

### 15.4 OpenAPI 설명과 실제 코드의 책임 경계가 길다

Rust `main.rs`와 Python `main.py` 안에 OpenAPI spec, 라우트, 비즈니스 로직, 렌더링 프록시, 보안 검증이 함께 들어 있다. 기능이 늘어나면 파일 단위 분리가 필요하다.

## 17. 결론

현재 시스템은 Open WebUI를 업무 포털처럼 사용하고, 뒤쪽에는 Rust/Python 기반 도구 서버를 붙인 문서 자동화 플랫폼이다. 구현의 중심은 구매 품의 문서 자동 생성과 재고 스냅샷이며, 내부 문서 검색과 PDF/Word/Excel 렌더링이 이를 보조한다.

실제 코드 수준에서는 이미 다음 기능이 구현되어 있다.

- 입고/재고/출고 Excel 기반 통합 스냅샷 생성
- 부족 재고 및 전체 재고 조회
- 품목별 구매 품의 컨텍스트 생성
- DOCX 템플릿 치환 방식의 구매 문서 생성
- 대량 DOCX 생성 및 ZIP 패키징
- Markdown 보고서 PDF 생성
- 채팅/보고서 DOCX, XLSX 내보내기
- 권한 기반 내부 문서 검색
- 외부 URL 본문 Markdown 추출

따라서 현재 아키텍처는 단순 챗봇이 아니라, “검색 → 근거 확보 → 문서 초안 → 승인 → 파일 산출”까지 이어지는 내부 문서 자동화 시스템으로 보는 것이 가장 정확하다.
